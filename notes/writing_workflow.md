# 写作工作流

## 1. 文件定位

本文件用于记录 Web GPT、作者、Codex、本地仓库、GitHub、Word/PDF 在毕业论文写作过程中的协同方式，以及每轮写作和修改任务应如何执行。

- `README.md`：仓库入口说明，帮助作者快速理解仓库用途和目录结构。
- `AGENTS.md`：Codex 行为规则，约束分支策略、修改边界、材料管理和禁止事项。
- `notes/writing_workflow.md`：每轮写作、整理、审阅和导出的执行流程说明。
- `notes/style_guide.md`：写作风格指南，用于后续沉淀中文学位论文表达规则。
- `notes/project_scope.md`、`notes/paper_to_thesis_mapping.md`、`notes/figure_registry.md`：后续写作控制文件，分别用于范围边界、TSTE 映射和图表证据链管理。

## 2. 总体原则

- GitHub `main` 是稳定版本，代表当前已审阅并合并的仓库状态。
- 本地仓库是 Codex 工作区，实际文件修改、commit、push 和导出测试都在本地完成。
- Markdown 是论文正文和说明文档的源文件。
- Word/PDF 是导出审阅稿和提交稿，不是正文权威来源。
- Web GPT 负责讨论、审阅、归纳问题、形成写作决策和整理 Codex 任务说明。
- 作者负责确认学术判断、范围边界、是否采用候选文本，以及是否 merge PR。
- Codex 只执行已确认的文件修改，不擅自扩写研究内容，不替作者做关键学术判断。
- 重要修改应通过分支、commit、push 和 Pull Request 审阅，避免直接在 `main` 上改动。

## 3. 标准闭环流程

1. Web GPT 读取 GitHub `main` 最新上下文。
2. Web GPT 判断当前任务类型和所需材料。
3. Web GPT 与作者讨论问题。
4. 作者确认修改方向、边界和禁止事项。
5. Web GPT 输出 Codex 可执行任务说明。
6. Codex 本地切换到 `main` 并 pull 最新版本。
7. Codex 创建任务分支。
8. Codex 按任务说明修改指定文件。
9. Codex 更新必要 notes，尤其是 `notes/change_log.md`。
10. Codex commit、push、创建 PR。
11. 作者审阅 diff。
12. 作者确认后 merge。
13. 本地仓库 pull 最新 `main`。
14. 下一轮从 `main` 开始。

这个闭环适用于文档说明、范围梳理、映射整理、图表登记、正文写作、润色和导出审阅。讨论可以发生在仓库外，但已确认决策应回写到仓库。

## 4. Web GPT 每轮固定事项

Web GPT 每轮需要做以下工作：

1. 判断当前任务类型。
2. 读取必要上下文。
3. 先审阅和讨论，不直接声称已经修改仓库文件。
4. 明确当前材料是否足够。
5. 明确本轮能做什么、不能做什么。
6. 明确需要作者确认的问题。
7. 给出修改建议或候选文本。
8. 区分“建议”“已确认决策”“需要 Codex 执行的任务”。
9. 最后整理成 Codex 可执行任务说明。

Web GPT 常见任务类型包括：

- 材料盘点
- 项目范围确认
- 小论文到毕设映射
- 图表证据链整理
- 章节结构设计
- 章节正文撰写
- 章节润色
- 结果分析强化
- 文献综述整理
- 导出稿审阅

Web GPT 的工作重点是讨论和判断，不是直接改本地仓库。它可以输出候选文字，但应明确哪些内容还需要作者确认，哪些内容已经可以交给 Codex 执行。

## 5. 作者每轮固定事项

作者每轮需要做以下工作：

1. 确认本轮任务目标。
2. 确认修改范围和禁止事项。
3. 确认学术边界。
4. 判断 Web GPT 给出的候选文本是否可采纳。
5. 决定是否交给 Codex 执行。
6. 审阅 Codex PR 或 diff。
7. 判断是否 merge。
8. 对最终论文质量和学术真实性负责。

作者是所有关键学术结论、论证边界、方法表述强度和最终论文质量的责任主体。Web GPT 和 Codex 都不能替代作者做最终确认。

## 6. Codex 每轮固定事项

Codex 每轮需要做以下工作：

1. 切换到 `main`。
2. 执行 `git pull --ff-only origin main`。
3. 创建符合任务类型的分支。
4. 检查任务是否有明确目标文件、允许修改文件和禁止修改文件。
5. 检查是否涉及正文、数据、图表、文献或结论。
6. 如果涉及关键学术内容，确认是否已有作者确认的写作决策。
7. 只修改任务指定范围。
8. Markdown 和 notes 内容非必要使用中文。
9. 更新必要 notes，尤其是 `notes/change_log.md`。
10. commit、push。
11. 如可行创建 PR。
12. 不自行 merge。
13. 最后报告修改文件、修改摘要、风险点和 PR/分支信息。

分支类型沿用 `AGENTS.md` 中的规则：

- `docs/`：说明文档、工作流、规则、notes 模板等文档修改
- `cleanup/`：目录清理、结构重组、无正文内容的整理任务
- `test/`：工具链、导出链路、格式检查等测试任务
- `draft/`：经作者确认后的论文草稿或正文相关写作任务
- `fix/`：修复错误、链接、格式、路径或说明不一致问题
- `export/`：生成或调整 Word/PDF 导出稿、导出脚本和导出配置

## 7. 不同任务类型的执行方式

### 7.1 材料盘点

Web GPT：

- 判断材料是否齐全
- 指出材料缺口
- 提出需要作者确认的问题

Codex：

- 生成或更新 `notes/material_inventory.md`
- 不分析技术结论
- 不写正文

### 7.2 项目范围确认

Web GPT：

- 讨论论文写什么、不写什么
- 明确 TSTE 与本科论文关系
- 明确 Dymola 状态
- 明确开放问题

Codex：

- 写入 `notes/project_scope.md`
- 更新 `notes/change_log.md`

### 7.3 小论文到毕设映射

Web GPT：

- 分析 TSTE 各部分如何转化
- 判断哪些内容重写、扩写、暂不使用
- 提出图表和公式候选

Codex：

- 写入 `notes/paper_to_thesis_mapping.md`
- 不直接生成正文

### 7.4 图表证据链整理

Web GPT：

- 判断每张图支撑什么结论
- 判断哪些图可用、需重画、待确认

Codex：

- 更新 `notes/figure_registry.md`
- 必要时整理 `assets/` 中的候选图表
- 不改原始仿真数据

### 7.5 章节结构设计

Web GPT：

- 基于 `project_scope`、`paper_to_thesis_mapping`、`figure_registry` 提出 5 章左右结构方案
- 说明每章功能、输入材料和输出目标

Codex：

- 创建 `thesis_source/chapters/` 下的章节 Markdown 模板
- 只写章节功能说明和占位
- 不直接写正文

### 7.6 正文写作

Web GPT：

- 先讨论本节写作目标
- 给出段落结构和候选文字
- 标注需要作者确认的内容
- 生成 Codex 任务说明

Codex：

- 将已确认内容写入指定 `chapter_XX.md`
- 进行局部润色和衔接
- 不扩展未确认内容
- 更新 `notes/change_log.md`
- 提交 PR

### 7.7 润色审阅

Web GPT：

- 基于 `notes/style_guide.md` 审阅语言、逻辑、段落衔接
- 指出问题并给修改建议

Codex：

- 执行局部润色
- 不改变技术含义
- 不新增数据或结论

### 7.8 导出审阅

Web GPT：

- 审阅导出稿中的结构、逻辑、语言和图表说明
- 提出内容性修改意见

Codex：

- 运行或维护导出脚本
- 生成 `exports/review/`
- 把内容性问题回写 Markdown
- 不把 Word 当正文源

## 8. Web GPT 每轮开场提示模板

以下模板用于作者在 Web GPT 新对话中启动任务：

```text
你现在先不要修改本地仓库文件。

当前任务类型：
[例如：材料盘点 / 项目范围确认 / 小论文到毕设映射 / 图表证据链整理 / 章节结构设计 / 正文写作 / 润色审阅 / 导出稿审阅]

请优先读取这些文件：
- [文件 1]
- [文件 2]
- [文件 3]

当前不要做的事情：
- 不要写论文正文 / 不要修改研究内容 / 不要编造数据 / 不要改动代码或图表 / 不要创建正式章节文件

本轮需要你输出：
1. 当前问题判断
2. 材料是否足够
3. 能做什么、不能做什么
4. 需要我确认的问题
5. 建议或候选文本
6. 最后整理成可直接交给 Codex 执行的任务说明
```

## 9. Codex 执行任务说明模板

以下模板用于作者把 Web GPT 已确认内容交给 Codex 执行：

```text
请基于当前本地仓库 main 最新版本执行以下任务。

任务类型：
[例如：docs / draft / fix / export / test / cleanup]

建议分支名：
[例如：docs/update-writing-workflow]

目标文件：
- [目标文件 1]
- [目标文件 2]

可修改文件：
- [允许修改文件 1]
- [允许修改文件 2]

禁止修改文件：
- [禁止修改文件 1]
- [禁止修改文件 2]

已确认内容：
- [已确认事项 1]
- [已确认事项 2]

待确认内容：
- [待确认事项 1]
- [待确认事项 2]

是否涉及正文：
[是 / 否]

是否需要更新 change_log：
[是 / 否]

是否需要 PR：
[是 / 否]

完成后请报告：
- 实际修改文件
- 修改摘要
- 风险点或待确认点
- commit 信息
- 分支名
- PR 链接
```

## 10. 审阅与 merge 规则

- 作者应先看 diff，再决定是否 merge。
- 审阅重点包括：是否改错文件、是否改变技术含义、是否编造内容、是否遗漏 `notes/change_log.md`。
- merge 后，本地仓库需要 pull 最新 `main`。
- 下一轮任务必须基于最新 `main` 开始。

这份工作流文档的目标不是替代 `AGENTS.md`，而是把后续毕业论文写作过程中的实际执行动作写清楚，便于长期反复使用。
