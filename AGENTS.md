# Codex 项目指导

## 1. 项目身份

- 项目类型：本科毕业论文协同写作仓库
- 论文题目：《商业建筑HVAC系统动态需求响应》
- 论文正文语言：中文；英文摘要除外
- 主要源文件格式：Markdown
- 导出格式：Word、PDF，仅用于审阅、阶段性检查和最终提交
- 远程事实源：GitHub `main` 分支
- 本地工作区：作者电脑中克隆的仓库文件夹，由 Codex 在本地修改并提交

本仓库用于支持毕业论文写作流程，而不是简单保存文件。核心工作流是：Web ChatGPT 负责讨论、审阅和形成写作决策；作者负责确认学术判断；Codex 负责在本地仓库中执行已确认的文件修改；GitHub 负责版本管理、分支、Pull Request 和历史追踪。

## 2. 总体工作流

每一轮实质性修改应遵循以下顺序：

1. Web ChatGPT 基于 GitHub `main` 的最新内容进行讨论和审阅。
2. 作者确认本轮修改目标、修改范围和禁止事项。
3. Codex 在本地仓库中切换到 `main` 并拉取远程最新版。
4. Codex 为本轮任务创建独立分支。
5. Codex 只修改本轮指定文件。
6. Codex 更新必要的 notes 文件，尤其是 `notes/change_log.md`。
7. Codex commit 并 push 分支到 GitHub。
8. 作者通过 Pull Request 或 diff 审阅变更。
9. 作者确认后再 merge 到 `main`。
10. 下一轮开始前，本地仓库再次 pull 最新 `main`。

Codex 不应自行 merge 重要 PR，除非作者明确要求。

## 3. 角色分工

### 作者

作者负责：

- 所有实质性学术判断；
- 确认研究范围、论点、结构和最终写作方向；
- 判断哪些内容可以写入毕业论文；
- 审阅 PR、diff、导出稿；
- 决定是否 merge；
- 对最终论文质量和学术责任负责。

### Web ChatGPT

Web ChatGPT 负责：

- 基于 GitHub 最新内容进行讨论和审阅；
- 帮助分析论文结构、逻辑、语言、证据链和审稿风险；
- 生成备选写法和关键段落候选文本；
- 提炼优秀论文和学校指南中的写作规则；
- 将讨论结果整理成可交给 Codex 执行的明确任务。

Web ChatGPT 不负责直接修改本地文件，也不替代 GitHub 版本管理。

### Codex

Codex 负责：

- 在本地克隆仓库中执行文件修改；
- 创建分支、修改 Markdown、更新 notes、commit、push；
- 整理文件、模板、笔记、图表登记和导出脚本；
- 根据已确认的写作决策进行局部润色、衔接和格式统一；
- 在需要时协助导出 Word/PDF 或记录导出失败原因。

Codex 不负责独立决定重大论文观点、贡献点、结论边界或实验解释。

### GitHub

GitHub 负责：

- 保存 `main` 稳定版本；
- 保存分支、commit、PR 和 diff；
- 作为 Web ChatGPT 和本地 Codex 之间的同步中心；
- 提供可追踪、可回滚的版本历史。

## 4. 语言和命名规则

- 文件夹名和文件名使用英文。
- 不使用拼音文件名。
- Markdown 正文内容使用中文，英文摘要、英文术语、代码、参考文献条目除外。
- `notes/` 中的写作决策、审阅意见、图表登记、变更记录尽量使用中文。
- commit message 可以使用英文，需简洁清楚。
- 论文标题、章节标题、图题、表题和正文表达应符合中文学位论文风格。

## 5. 文件组织规则

推荐的简化仓库结构如下：

```text
requirements/        学校指南和模板
style_samples/       直接存放可参考的优秀硕博论文样本
journal_paper/       IEEE TSTE 小论文材料，统一存放
research_materials/  仿真 cleanroom、Dymola 材料和仿真记录
thesis_source/       毕业论文 Markdown 源文件
assets/              最终入文图表、候选图表和图源文件
references/          references.bib、中文文献候选、引用说明
notes/               项目范围、流程、映射、图表登记、决策、审阅、日志
exports/             审阅稿和提交稿
scripts/             导出、图表处理、辅助检查脚本
archive/             暂时不用但不确定是否可删除的历史文件
```

`research_materials/simulation/` 应作为混合语言仿真项目的整体存放位置，后续导入时保持项目内部 `run/`、`export/`、`plot/`、`src/`、`data/`、`docs/` 等结构，不再按编程语言拆分。Dymola 平台或模型材料单独放入 `research_materials/dymola/`。

如果仓库中存在测试阶段留下的临时文件或过细目录，Codex 可以在作者确认后提出整理方案，但不要擅自删除原始材料。

## 6. Markdown 与 Word/PDF 的关系

- `thesis_source/` 下的 Markdown 是论文内容源文件。
- `exports/` 下的 Word/PDF 是导出稿，不是权威源文件。
- 从 Word/PDF 中发现的内容问题，应回流到 Markdown 中修改。
- 不要在 Word 中长期维护另一套正文版本。
- 纯格式问题可以在最终 Word 阶段处理，但内容性修改必须回到 Markdown。

## 7. 禁止编造规则

Codex 不得编造：

- 数据；
- 数值；
- 图表；
- 参考文献；
- 实验结果；
- 仿真输出；
- 引文；
- 学术观点；
- 文献结论。

如果信息缺失，Codex 应保留占位、模板或明确的开放问题，而不是补写不存在的内容。

## 8. 写作边界

- 不得将 IEEE TSTE 小论文直接翻译成中文毕业论文正文。
- 可以在明确要求下帮助整理、比较、映射、概括和改写材料。
- 关键段落、贡献点、结果解释、结论边界应依据 Web ChatGPT 与作者确认后的写作决策。
- 不得擅自改变已有实验数值、图表含义或结论强度。
- 对不确定内容，应标注为“待确认”，不得自行填充。
- Dymola 平台部分目前仅作为材料和可能的说明内容，不得擅自表述为最终方法验证，除非作者确认。

## 9. 分支、commit 与 PR 规则

### 开始任务前

Codex 应先执行或等价完成：

```bash
git switch main
git pull --ff-only origin main
git switch -c <branch-type>/<short-task-name>
```

### 分支规则

- 每个分支只解决一个明确任务。
- 正文修改、结构调整、规则更新、图表登记、参考文献调整应优先使用独立分支。
- 不要在 `main` 上直接进行实质性写作或结构修改。
- 分支类型应根据任务性质选择：
  - `docs/`：说明文档、工作流、规则、notes 模板等文档修改；
  - `cleanup/`：目录清理、结构重组、无正文内容的整理任务；
  - `test/`：工具链、导出链路、格式检查等测试任务；
  - `draft/`：经作者确认后的论文草稿或正文相关写作任务；
  - `fix/`：修复错误、链接、格式、路径或说明不一致问题；
  - `export/`：生成或调整 Word/PDF 导出稿、导出脚本和导出配置。

### commit 规则

- commit 范围应清晰。
- commit message 应说明改了什么。
- 不要把无关修改混在一个 commit 中。

### PR 规则

- 实质性写作变更应优先创建 Pull Request。
- PR 描述应包括：修改目标、修改文件、主要变化、需要作者审阅的风险。
- Codex 不应自行 merge 重要 PR。

## 10. 变更记录规则

重要变更后应更新 `notes/change_log.md`，记录：

- 日期；
- 分支或任务名；
- 变更类型；
- 是否涉及正文；
- 修改文件；
- 修改摘要；
- 是否已 push / PR；
- 是否需要作者审阅；
- 后续事项。

已确认的写作或结构决策，应记录到 `notes/writing_decisions.md`。Web ChatGPT 或人工审阅意见如果会影响后续工作，应记录到 `notes/review_comments.md`。

## 11. 每次 Codex 执行任务前的固定检查

Codex 在执行任务前应检查：

1. 当前任务是否有明确目标文件。
2. 当前任务是否有明确禁止修改范围。
3. 是否已经基于最新 `main` 创建分支。
4. 是否涉及正文、数据、图表、文献或结论。
5. 如果涉及关键学术内容，是否已有作者确认的写作决策。
6. 是否需要更新 `notes/change_log.md`。
7. 是否需要 push 和创建 PR。

如果任务说明不清楚，Codex 应先提出需要确认的问题，而不是猜测执行。

## 12. 模型使用原则

为了节省额度并保证效率，Codex 可根据任务复杂度选择模型：

- 文件整理、建目录、移动文件、更新模板：使用低成本或快速模型。
- 写入 Web ChatGPT 已确认的文字：使用低到中等模型。
- 局部中文润色、过渡句、段落衔接：使用中等模型。
- 整章重构、结果解释、贡献点相关内容：应先由 Web ChatGPT 与作者定稿，再由 Codex 使用中高模型执行。
- Git 冲突、导出失败、跨文件一致性检查：可使用中高模型。
- 最终全文一致性、术语统一、引用与图表检查：建议使用中高模型。

Codex 不应为了简单文件操作消耗高额度模型。

## 13. 删除和重组规则

- 不得擅自删除原始数据、原始代码、小论文材料、学校指南、样本论文或参考文献。
- 测试文件、临时导出文件、空目录、无实际用途的占位文件可以在作者确认后删除。
- 如果需要重组仓库结构，应先输出重组计划，说明：保留什么、移动什么、删除什么、为什么。
- 对不确定是否可删除的文件，应移动到 `archive/` 或列入待确认清单，而不是直接删除。
